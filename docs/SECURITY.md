# PharmaFlow Enterprise Security Architecture Blueprint

توضح هذه الوثيقة المعمارية الأمنية لمشروع **PharmaFlow ERP**، وتحديداً التحليق الهندسي لمنع تسريب المفاتيح السرية وحماية البنية التحتية المحاسبية والمخزنية وفقاً لأعلى معايير الأمن السيبراني (OWASP & SOC2 compliant).

---

## 1. مخاطر بادئة `VITE_` على المفاتيح السرية (The client-side exposure threat)

### المشكلة الأمنية (The Vulnerability)
في الأنظمة التي تعتمد على المجمعات الحديثة للواجهات الأمامية مثل Vite، يتم التعامل مع متغيرات البيئة تلقائياً طبقاً لتسميتها:
* أي متغير يبدأ بالبادئة `VITE_` (مثل `VITE_ENCRYPTION_KEY` أو `VITE_SECRET`) يتم **حقنه مباشرة وبشكل مرئي** داخل ملفات كود المتصفح المجمع (JS Bundles).
* يمكن لأي متسلل أو عميل بسيط فتح أدوات المطورين (DevTools Core) والذهاب إلى علبة الذاكرة أو ملفات الجافا سكربت المجمعة واستخراج المفاتيح أو نصوص التشفير بكل سهولة.

### الحل المطبق (Our Solution)
تم التخلص نهائياً وبتر كافة المتغيرات التي تبدأ بـ `VITE_` بخصوص التشفير من الواجهة الأمامية، وتم نقل المفتاح بالكامل إلى خادم الخدمة الخلفي (Backend Node.js Core) تحت اسم:
`ENCRYPTION_KEY`
والذي لا تقوم حزمة Vite بقراءته أو عرضه أو حزمته مطلقاً للواجهة الأمامية.

## 2. معمارية التشفير الخلفي (Backend Cryptography Engine)

يعمل محرك التشفير في تطبيقنا خلفياً بالكامل عبر المسار `server/security/encryption.service.ts` باستخدام خوارزمية **AES-256-GCM** (Advanced Encryption Standard with Galois/Counter Mode):

```
+-------------------------------------------------------------+
|                     Client Browser                          |
|  (Never holds keys. Requests encrypt/decrypt as proxy APIs) |
+------------------------------+------------------------------+
                               |
                        Secure HTTPS Request
                               |
                               v
+------------------------------+------------------------------+
|                        Express server                       |
|  - Rate limited (max 60 req/min for security operations)     |
|  - Helmet CSP headers enabled                               |
|  - Runs scan-secrets.js statically at compilation time      |
+------------------------------+------------------------------+
                               |
                               v
+------------------------------+------------------------------+
|               Node.js Crypto (Private Core)                 |
|  - Loads ENCRYPTION_KEY (strictly fails if absent)           |
|  - Symmetric 256-bit hashing (crypto.pbkdf2Sync & sha256)   |
|  - Generates cryptographically secure random IV (12 bytes)   |
+-------------------------------------------------------------+
```

### مميزات خوارزمية AES-256-GCM:
1. **السرية التامة (Confidentiality):** تشفير 256-بت آمن ضد هجمات القوة الغاشمة (Brute-Force).
2. **سلامة البيانات والتأكد الذاتي (Data Integrity & Authenticity):** يولّد كود توقيع للمصادقة (`authTag`). أي تلاعب أو تعديل ولو ببايت واحد في الملف المشفر سيفشل كلياً في الفك ولن يتم قبوله، مما يمنع هجمات الحشو التشفيرية (Padding Attacks).
3. **التشويش العشوائي (Entropy via Random IVs):** في كل عملية تشفير، يتم توليد ناقل تهيئة عشوائي (`iv`) بحجم 12 بايت، مما يعني أن تشفير نفس النص مرتين سينتج عنه مخرجات مشفرة مختلفة تماماً في كل مرة.

## 3. خطة تدوير المفاتيح الأمنية (Key Rotation Strategy)

في بيئات الأعمال والإنتاج الحقيقية، يُنصح بتدوير المفتاح (`ENCRYPTION_KEY`) بصفة دورية (مثلاً كل 90 يوماً أو فور الاشتباه في تسريب أمني):

```bash
# 1. قم بفك تشفير النسخ الاحتياطية الحالية باستخدام المفتاح القديم
# 2. قم بتوليد مفتاح تشفير جديد عالي الكثافة:
node fix-security.js

# 3. قم بإعادة تشفير النسخ التي تم فكها باستخدام المفتاح الجديد
# 4. حدّث متغيرات البيئة في السحابة وأعد تشغيل الخادم
```

## 4. أفضل الممارسات الأمنية (OWASP Best Practices Implemented)

* **التحقق الصارم من المدخلات (Strict Input Filtering & Validation):** يضمن المحرك عدم تمرير قوالب خاطئة أو ملفات خبيثة قد تسبب فرط استخدام الذاكرة أو إنهاء الخدمة.
* **البناء الآمن والفحص الساكن (Secure Build Pipeline):** تم حقن الأداة `scripts/security/scan-secrets.js` داخل نظام التشغيل العام، وتعمل تلقائياً لمنع أي عملية تجميع للإنتاج (Build) إذا تم كشف وجود كلمات حساسة مثل `VITE_SECRET` أو غيرها داخل أسلاك الكود.
* **الكبسلة ضد الأخطاء الحساسة (Error Boundaries):** تم كبسلة وتطويق كافة دوال الأخطاء للعملاء؛ لا يتم أبداً إرجاع الـ Stack Traces أو المتغيرات الداخلية في الـ Response في حال حدوث خطأ، بل يظهر خطأ معرّف معقم (Sanitized Operations Error).
