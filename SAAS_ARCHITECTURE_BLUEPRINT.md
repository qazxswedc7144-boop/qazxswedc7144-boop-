# مخطوطة بنية الأنظمة السحابية للشركات والربط الطبي | SaaS Enterprise Architecture Blueprint
### PharmaFlow Pro ERP — Hybrid Healthcare Interoperability Standard

تضم هذه المخطوطة المواصفات الفنية والهندسة المعمارية لترقية وتفعيل بنية **SaaS (برمجيات كخدمة)** متكاملة لربط الصيدليات بالمؤسسات الصحية الكبرى، والمستشفيات، ومعايير الترابط الصحي العالمية والمحلية في المملكة العربية السعودية (مثل منصة **نفات** ومنصة **صحة**).

---

## 🏗️ 1. الهندسة المعمارية متعددة المستأجرين (Multi-Tenant Architecture)

يتبع تطبيق **PharmaFlow Pro** نموذج عزل معماري هجين يجمع بين الأمان الفصلي والسرعة التشغيلية الفائقة:

```
               [ بوابة مطوري المؤسسات / API Gateway ]
                                |
       _________________________|_________________________
      |                                                   |
[ المستأجر 1: مجموعة دلة الطبية ]             [ المستأجر 2: مستشفى المواساة ]
      |                                                   |
      |-- (عزل منطقي / تشفير معزول)                     |-- (عزل منطقي / تشفير معزول)
      |-- معيار قاعدة البيانات: Tenant Schema             |-- معيار قاعدة البيانات: Tenant Schema
      |-- مفتاح تشفير فريد: AES_Dallah_2026               |-- مفتاح تشفير فريد: AES_Mouwasat_2026
```

### أ. مستويات العزل (Isolation Levels)
1. **العزل المنطقي (Logical Isolation)**: تشارك جميع المؤسسات نفس البنية التشغيلية، ولكن تحمل كل كتل البيانات حقلاً غير قابل للتعديل لمعرف المستأجر `tenantId`.
2. **التشفير المنفصل للمستأجرين (Tenant-Key Vault Isolation)**: تشفير الحقول الحيوية (أسماء المرضى والأرقام القومية وبيانات الفواتير المالية) بمفتاح تشفير متولد خصيصاً لكل مستأجر عبر PBKDF2 والـ Salt الملحق بمعرف المستأجر.

### ب. العزل على مستوى قواعد البيانات المحلية (IndexedDB / Dexie.js Isolation)
في مستوى الواجهة الأمامية، يتم تزويد كافة فئات الوصول والمستودعات (Repositories) بمسارات تصفية وتفتيش تمنع اختلاط كاش البيانات محلياً:
```typescript
// ضمان تصفية السجلات آلياً بهوية المستأجر النشط لمنع المزامنة المشتركة
export const getActiveTenantInvoices = async (db: any, tenantId: string) => {
  return await db.invoices
    .where('tenantId')
    .equals(tenantId)
    .toArray();
};
```

## 🔒 2. المزامنة المشفرة والتأمين السحابي (Encrypted Cloud Sync)

يستخدم النظام دورة مزامنة سحابية فائقة الأمان لترحيل بيانات الصيدليات والمخازن إلى مراكز البيانات السحابية المركزية:

### أ. دورة ترحيل القيود (Replication Loop)
1. **شحن البيانات (Data Preparation)**: يجمع التطبيق السجلات المحلية غير المزمنة.
2. **التشفير الطرفي الحصين (End-to-End Encryption)**:
   - يتم اشتقاق مفتاح تشفير 256 بت خاص بالمؤسسة باستخدام معيار PBKDF2.
   - يتم تشفير الحمولة الإجمالية باستخدام خوارزمية **AES-GCM** مع إضافة ناقل تشخيصي (Initialization Vector - IV).
3. **النقل الآمن (Secure Transit)**: تبعث الحمولة المشفرة عبر بروتوكول TLS 1.3 الآمن إلى ملقم REST API بالخلفية.
4. **التحقق من التماسك (Integrity Handshake)**: يوثق الخادم سلامة الملفات عن طريق مقارنة بصمة SHA-256 المرفقة للتأكد من عدم العبث بالبيانات أثناء النقل.

## 🏥 3. التكامل الطبي والترابط بمعايير HL7 FHIR
يدعم نظام **PharmaFlow Pro** التوافق البيني بالكامل لربط الصيدليات بشبكات الرعاية الصحية الخارجية والـ EMR للمستشفيات عبر معيار **HL7 FHIR R4** الدولي:

```
[ نظام مستشفى EMR ] ----(MedicationRequest R4 PDF/JSON)----> [ بوابة PharmaFlow Interop Gateway ]
                                                                       |
                                                               (FHIR R4 Parser)
                                                                       |
                                                         [ ترحيل فوري لمبيعات الصيدلية ]
```

### أ. هيكلة كائنات وموارد FHIR (FHIR Resources Supported)
- **Patient Resource**: تمثيل بيانات المريض، أرقام الهواتف، الهوية الوطنية المعتمدة.
- **Medication Request Resource**: قالب الوصفات الطبية المحولة من الطبيب المعالج، الجرعات، وتاريخ انتهاء الصلاحية المأذون به.
- **Organization Resource**: بيانات المستشفى المصدر والجهة الطبية المشرفة على الصيدلية.

## 🔌 4. بوابة المطورين ومفاتيح الاتصال البرمجي (SaaS REST API Gateway)

توفر البوابة ملقماً واعداً للمبرمجين ومهندسي الأنظمة في المستشفيات لإتمام طلبات الصرف الفوري والتحقق من النواقص البرمجية:

### أ. عناوين الملقم المعيارية (API Endpoints Roadmap)

| المسار (Endpoint) | الطريقة (Method) | الصلاحية المطلوبة (Scope) | الوصف العلمي والتقني |
| :--- | :--- | :--- | :--- |
| `/api/v1/saas/fhir/Patient` | `GET` | `fhir.read` | سحب قائمة المرضى النشطين ببيانات متوافقة مع FHIR R4 |
| `/api/v1/saas/fhir/MedicationRequest` | `POST` | `fhir.write` | تقديم وصفة طبية إلكترونية جديدة مرسلة من المستشفى للمطابقة والصرف |
| `/api/v1/saas/sync` | `POST` | `financials.read` | إرسال حزمة المزامنة المشفرة محلياً بالكامل من الصيدليات للسحابة |

### ب. نظام المفاتيح وقبعة الصلاحيات الإضافية (Scoped API Tokens)
تبدأ كافة مفاتيح الربط التي يولدها النظام بالبادئة المعيارية `pf_live_` لمنع الخطأ في كود الأمان وتستخدم الصلاحيات المقيدة (Scopes) لحساب الصلاحيات بدقة متناهية:
* `fhir.read`: قراءة الملفات الطبية للمريض.
* `fhir.write`: إدراج وصرف فوري للوصفات الطبية.
* `inventory.read`: فحص المتبقي من الأدوية بمخزن الصيدلية لمنع إرشاد ووصف دواء غير متوفر.

## 📋 5. الامتثال والتدقيق القانوني للرعاية الصحية (Compliance & Regulation Audit)

تم تصميم النظام ليكون متوافقاً بشكل صارم مع التشريعات القانونية الحيوية لرعاية وسجل البيانات الطبية والمالية:

1. **الامتثال لـ نظام حماية البيانات الشخصية السعودي (PDPL)**:
   - تشفير وسرية بيانات المريض والبيانات الائتمانية بشكل كامل.
   - منع سحب أي بيانات طبية حساسة خارج النطاق الجغرافي للمملكة العربية السعودية دون تشفير ثنائي الاتجاه وتفويض أمني معتمد.
2. **الامتثال للائحة معايير HIPAA (قانون نقل التأمين الصحي والمساءلة)**:
   - عزل تام لكل مستأجر.
   - سجلات مراجعة حيوية (Audit Log) ترصد نشاط المستخدمين ومفاتيح API بالتوقيع بالثواني والملي ثانية.
3. **معيار ISO/IEC 27001 لسلامة وأمن المعلومات صيدلانياً ومؤسسياً**:
   - حجب قنوات التدخل اليدوي بالـ Database وتعزيز سجلات الفواتير الموثقة المقفلة لمنع تعديل القيود المالية بعد الصرف والاعتماد (Period Locking).
